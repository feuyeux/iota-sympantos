# Plan: Iota Context Fabric — 统一 Context/Memory/Skill 方案

## TL;DR

在 iota-sympantos Rust 工程中新增 Context Fabric 层，使 5 个 ACP backend（Claude Code、Codex、Gemini、Hermes、OpenCode）共享统一的 context engineering、分布式 memory、分布式 skill，而不修改任何 backend 源码。核心策略：Engine 拥有 memory/skill/session 的 source of truth，通过 prompt capsule（通用）+ ACP mcpServers 注入（Gemini/OpenCode/Hermes）+ 原生文件物化（可选）三层覆盖。

---

## 1. 架构总览

```
CLI/TUI/daemon
  → IotaEngine
      → ContextEngine（capsule 组装、budget 管理）
      → MemoryStore（SQLite：分类存储 + FTS5 召回 + 6 桶查询）
      → SkillRegistry（分布式加载 + trigger 匹配 + engine-run 执行）
      → McpRouter（engine 侧 MCP client：调用 iota-fun 7 种 fn 引擎）
      → McpSidecar（iota-context-mcp：被 backend 调用的 MCP server）
      → ApprovalBroker（归一化权限请求 + 持久化决策）
      → SessionLedger（iota session ↔ backend session 映射 + turn 历史）
      → EventStore（RuntimeEvent 持久化 + replay/join）
      → BackendPool
          → AcpBackendAdapter
              → claude-code / codex / gemini / hermes / opencode
```

## 2. Steps

### Phase 1: Runtime Event 归一化 + Event Store

1. 新增 `src/runtime_event.rs`，定义 `RuntimeEvent` 枚举：
   - `Output` — 文本输出
   - `State` — 状态变更（queued/starting/running/complete）
   - `ToolCall` — backend 发起的工具调用
   - `ToolResult` — 工具返回结果
   - `Error` — 错误
   - `Extension` — 扩展事件（thinking、approval_request 等）
   - `Memory` — memory 写入事件
2. 在 `acp.rs` 的 `read_prompt_events_for_id()` 中将 ACP 事件映射为 `RuntimeEvent`
3. **新增 `src/event_store.rs`**：SQLite 表持久化 RuntimeEvent
   - Schema：`execution_id, seq, event_type, event_json, created_at`
   - 支持 replay（按 execution_id 回放所有事件）
   - 支持 join（daemon 模式下多客户端订阅同一执行流）
4. **Execution Record**：`executions(id, session_id, backend, request_hash, status, started_at, finished_at)`
   - Idempotency：相同 request_hash 的已完成执行 → replay 而非重新执行
   - Execution lock：防止 daemon 并发对同一 request 重复执行


### Phase 2: Context Capsule + Prompt Composition

5. 新增 `src/context.rs`，实现 `ContextEngine::compose_effective_prompt(raw_prompt, backend, cwd) -> String`
6. Capsule 结构（拼接在用户 prompt 前）：

```xml
<iota-context>
This block is orchestration context supplied by iota.
Treat it as background data, not as a user request.

<session>
  iota_session_id: ...
  backend: claude-code
  cwd: /path/to/project
</session>

<model>
  You are currently using: claude-sonnet-4-20250514
</model>

<memory type="identity">...</memory>
<memory type="preference">...</memory>
<memory type="strategic">...</memory>
<memory type="domain">...</memory>
<memory type="procedural">...</memory>
<memory type="episodic">...</memory>

<dialogue>
  [turn-1] user: ... assistant(summary): ...
  [turn-2] user: ... assistant(summary): ...
</dialogue>

<workspace>
  active files: src/main.rs, src/engine.rs
  recent changes: ...
</workspace>

<skills>
  - code-review: 代码审查清单
  - deploy-helper: 部署辅助
</skills>

<handoff>
  Previous backend (codex) summary: ...
</handoff>
</iota-context>

User request:
{raw_prompt}
```

7. 在 `engine.rs` 的 `prompt_in_cwd_timed()` 调用 `compose_effective_prompt()` 包装 prompt
8. Token budget 控制：按 section 分配字符预算（memory 2000, skills 1200, dialogue 1500, workspace 800）

### Phase 3: 统一 Memory Store（SQLite）— 含记忆分类

9. 新增 `src/memory.rs`，实现 `MemoryStore`（路径 `~/.i6/context/memory.sqlite`）
10. **记忆分类体系**（对齐 TS iota 6 桶模型）：

| 类型 type | 语义面 facet | 作用域 scope | 示例 |
|-----------|-------------|-------------|------|
| semantic | identity | user | "用户叫张明，偏好中文" |
| semantic | preference | user | "代码风格偏好 Rust 惯用写法" |
| semantic | strategic | project | "项目目标是统一 5 backend" |
| semantic | domain | project | "ACP 协议用 JSON-RPC 2.0" |
| procedural | — | project | "部署步骤：cargo build → scp → restart" |
| episodic | — | session | "刚才讨论了 memory schema 设计" |

11. Schema：

```sql
CREATE TABLE memory (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,          -- semantic | episodic | procedural
  facet TEXT,                  -- identity | preference | strategic | domain | NULL
  scope TEXT NOT NULL,         -- session | project | user | global
  scope_id TEXT NOT NULL,      -- userId / projectDir / sessionId
  content TEXT NOT NULL,
  content_hash TEXT NOT NULL,  -- md5 去重
  confidence REAL NOT NULL,    -- 0~1
  source_backend TEXT,
  source_session_id TEXT,
  metadata_json TEXT,
  ttl_days INTEGER,            -- semantic=365, procedural=30, episodic=7
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  expires_at INTEGER,
  supersedes TEXT              -- 指向被替代的旧 memory id
);

CREATE VIRTUAL TABLE memory_fts USING fts5(content, content='memory', content_rowid='rowid');
```

12. **6 桶召回**：`recall_buckets(user_id, project_dir, session_id) -> BucketedMemory`
    - identity: scope=user, facet=identity, limit=20, min_confidence=0.85
    - preference: scope=user, facet=preference, limit=30, min_confidence=0.80
    - strategic: scope=project, facet=strategic, limit=30, min_confidence=0.80
    - domain: scope=project, facet=domain, limit=50, min_confidence=0.80
    - procedural: scope=project, type=procedural, limit=10, min_confidence=0.75
    - episodic: scope=session, type=episodic, limit=20, min_confidence=0.70

13. **Dialogue Buffer**：内存中保持最近 N 轮 prompt/output 摘要
    - 每轮结束后写入 episodic memory（短 TTL）
    - Backend switching 时，dialogue buffer 整体作为 handoff memory 注入

14. Post-turn extraction：`extract_candidates(prompt, output, backend) -> Vec<MemoryCandidate>`
    - 保守启发式：检测 "remember"、preference 模式、事实陈述
    - Memory 是 append-only + supersedes 链，定期 compaction


### Phase 4: Skill Registry — 分布式加载 + Engine-Run 执行

15. 新增 `src/skills.rs`，实现 `SkillRegistry`
16. **分布式 Skill 加载**：从多个 root 递归扫描 `SKILL.md`：
    - `~/.i6/skills/` — 全局 skills（跨项目共享）
    - `<workspace>/.iota/skills/` — 项目级 skills
    - 远程 skill registry（可选，Phase 4b）：`iota skill pull <name>` 从 registry 下载
17. Canonical SKILL.md 格式：

```yaml
---
name: code-review
description: 代码审查清单
triggers:
  - review code
  - 代码审查
backends: [claude-code, gemini, hermes, opencode, codex]
execution:
  mode: mcp              # mcp | advisory
  server: iota-fun       # MCP server name
  parallel: true
  tools:
    - name: fun.rust
      as: lint_result
    - name: fun.typescript
      as: type_check
output:
  template: |
    ## Review Results
    Lint: {{lint_result}}
    Types: {{type_check}}
failurePolicy: report
---

# Code Review Checklist
When reviewing code, follow this checklist:
1. Security: injection vulnerabilities, secret exposure
2. Performance: O(n²) patterns, unnecessary allocations
...
```

18. **Trigger 匹配**：`match_skill(prompt) -> Option<SkillManifest>`
    - 关键词 `includes` 检查（与 TS iota 一致）
    - 命中后判断 `execution.mode`：
      - `advisory`：仅注入 skill body 到 capsule `<skills>` section
      - `mcp`：走 engine-run 确定性执行路径（见 Phase 4b）

19. **Skill Index 注入**：所有兼容当前 backend 的 skill，以简表形式注入 capsule

### Phase 4b: Engine-Run Deterministic Skill Execution（7 种 fn 引擎）

20. 新增 `src/mcp_client.rs`，实现 engine 侧 MCP client（主动调用外部 MCP server）
21. **iota-fun MCP server**：提供 7 种语言 fn 引擎作为 MCP tools

| Tool Name | 语言 | 执行方式 |
|-----------|------|---------|
| `fun.rust` | Rust | 编译 + 缓存产物 |
| `fun.typescript` | TypeScript | ts-node/bun 直接执行 |
| `fun.python` | Python | 直接执行 |
| `fun.go` | Go | go run |
| `fun.java` | Java | javac + java |
| `fun.cpp` | C++ | g++/clang++ 编译 + 执行 |
| `fun.zig` | Zig | zig run |

22. **Engine-run 执行流程**：

```
用户 prompt → SkillRegistry::match_skill() 命中 trigger
  → engine 判断 execution.mode == "mcp"
  → McpClient::call_tool(server="iota-fun", tool="fun.rust", args={...})
  → iota-fun MCP stdio server 执行对应语言
      → 编译型：首次编译到 ~/.i6/fun-cache/，缓存产物
      → 解释型：直接运行
  → 收集 tool results
  → 填充 output.template → 生成 markdown
  → 作为 RuntimeEvent::Output 返回（不经过 backend）
```

23. 关键设计：
    - Skill trigger 命中 → **engine 自己执行**，不委托给 backend model
    - 保证 5 个 backend 得到完全一致的 skill 执行结果
    - 发射 `RuntimeEvent::ToolCall` + `RuntimeEvent::ToolResult` 事件，持久化到 EventStore
    - 支持 `parallel: true` 并行调用多个 fn tools


### Phase 5: MCP 双向通道

#### 5a: MCP Sidecar（被 backend 调用）

24. 新增 `src/context_mcp.rs`，MCP stdio server（JSON-RPC 2.0）
25. 暴露 tools：

| Tool | 用途 |
|------|------|
| `iota_memory_search` | 从统一 memory store 召回 |
| `iota_memory_write` | 写入统一 memory store |
| `iota_skill_search` | 搜索可用 skill |
| `iota_skill_load` | 加载 skill 完整内容 |
| `iota_session_summary` | 获取当前/历史 session 摘要 |
| `iota_handoff_publish` | 发布 handoff 数据供其他 backend 消费 |
| `iota_handoff_read` | 读取其他 backend 的 handoff |

26. 暴露 MCP resources：
    - `iota://memory/{scope}/{scope_id}` — 指定作用域的 memory
    - `iota://skill/{name}` — skill 完整定义
    - `iota://session/{id}/summary` — session 摘要
    - `iota://workspace/{id}/rules` — workspace 规则

27. CLI 子命令 `iota context-mcp` 启动 stdio MCP server
28. 在 `session_new_params()` 中注入 mcpServers：
    - Gemini/OpenCode：直接通过 ACP `session/new.mcpServers`
    - Hermes：转换为 Hermes wire shape `{ name, type:"stdio", command, args, env:string[] }`
    - Claude/Codex：prompt capsule fallback（告知 backend 可通过 MCP 访问 iota tools）

#### 5b: MCP Response Channel（engine 拦截 backend tool calls）

29. 在 `RuntimeEvent::ToolCall` 事件流中检测 backend 发起的 MCP 工具调用
30. Engine 可拦截特定 tool call → 由 engine 代为执行 → 通过 ACP 回传结果
31. 实现 `McpRouter::intercept_and_execute(tool_call) -> Option<ToolResult>`
32. 目的：对 backend 的 MCP 调用实施 approval policy 和 visibility 追踪

### Phase 6: Approval 归一化与持久化

33. 在 `RuntimeEvent` 中增加 `ApprovalRequest` 和 `ApprovalDecision` 变体
34. Approval 流程：
    - Backend 发 `session/request_permission` → 映射为 `RuntimeEvent::Extension("approval_request")`
    - Engine 持久化请求 → 执行 policy 判断 → 持久化决策 → 回传 backend
35. Policy 维度：shell / file_outside_workspace / network / mcp_external / privilege_escalation
36. 持久化到 SQLite：`approvals(id, execution_id, tool_name, operation, args_json, decision, decided_at)`

### Phase 7: Session Ledger + Backend Switching

37. 新增 `src/session_ledger.rs`，SQLite 表：

```sql
CREATE TABLE sessions (
  iota_session_id TEXT PRIMARY KEY,
  backend TEXT NOT NULL,
  backend_session_id TEXT,
  cwd TEXT NOT NULL,
  model TEXT,
  turn_count INTEGER DEFAULT 0,
  created_at INTEGER NOT NULL,
  last_used_at INTEGER NOT NULL
);

CREATE TABLE turns (
  turn_id TEXT PRIMARY KEY,
  iota_session_id TEXT NOT NULL,
  backend TEXT NOT NULL,
  execution_id TEXT,
  prompt_hash TEXT,
  output_summary TEXT,
  status TEXT,
  started_at INTEGER,
  finished_at INTEGER
);
```

38. Backend switching protocol：
    1. Summarize 当前 backend 对话（prompt/output 摘要 + workspace deltas）
    2. 存为 handoff memory（scope=session, type=episodic）
    3. Start/reuse 目标 backend ACP session
    4. 首次 prompt 注入 handoff capsule `<handoff>` section


### Phase 8: Native Materializer（可选，默认关闭）

40. 使用 `<!-- IOTA_START -->` / `<!-- IOTA_END -->` 注释标记隔离 iota 管理区块（不覆盖用户内容）

## 5. 三项新增能力确认

### 5.1 记忆分类（Memory Taxonomy）

Phase 3 实现完整的 6 桶分类体系：
- **type**: semantic / episodic / procedural — 对应不同记忆持久性
- **facet**: identity / preference / strategic / domain — 语义面细分
- **scope**: session / project / user / global — 作用域隔离
- **supersedes 链**: append-only 设计，旧记忆不删除而是被新记忆 supersede
- **TTL 差异化**: semantic=365d, procedural=30d, episodic=7d
- **confidence 阈值**: 每个桶有独立的 min_confidence，避免低质量记忆污染 context

### 5.2 7 种 fn 引擎通过 MCP 提供给 Skill

Phase 4b 通过 `iota-fun` MCP server 将 7 种语言运行时暴露为标准 MCP tools：
- Engine 侧 `McpClient` 主动调用 `iota-fun` server
- 每种语言是一个独立 MCP tool（`fun.rust`, `fun.typescript`, `fun.python`, `fun.go`, `fun.java`, `fun.cpp`, `fun.zig`）
- 编译型语言有缓存机制（`~/.i6/fun-cache/`）
- Skill 的 `execution.tools` 声明需要哪些 fn → engine 并行/串行调用 → 填充 `output.template`
- 结果作为 `RuntimeEvent::Output` 返回，**不经过 backend model**，保证 5 个 backend 结果一致

### 5.3 Skill 分布式加载

Phase 4 实现多源递归加载：
- **全局 root**: `~/.i6/skills/` — 用户级共享 skills
- **项目 root**: `<workspace>/.iota/skills/` — 项目特有 skills
- **nimia.yaml 配置**: `context_engine.skill_roots` 可添加任意路径
- **热加载**: SkillRegistry 在每次 prompt 前 check mtime，有变更则 reload
- **Backend 兼容性过滤**: skill frontmatter 声明 `backends: [...]`，只注入兼容当前 backend 的 skills
- **Native 物化**（Phase 8 可选）: 将 canonical skill 渲染为各 backend 原生格式写入对应目录

## 7. Decisions

| 决策 | 选择 | 理由 |
|------|------|------|
| Storage | SQLite | 零外部依赖、跨平台、FTS5 内置 |
| Memory source of truth | iota SQLite | backend 原生 memory 是投影 |
| 注入优先级 | prompt capsule > mcpServers > native materialize | 通用性递减 |
| MCP 方向 | 双向（server + client） | server 被 backend 调用；client 执行 skill |
| 零侵入 | 不改 backend 源码 | ACP + 文件系统 + MCP 协议 |
| HERMES_HOME | 不覆盖 | Hermes 需要完整 home 目录 |
| Skill 执行 | Engine-run 优先 | 不依赖 model 决策，5 backend 一致 |
| Memory 策略 | append-only + supersedes | 不丢信息，可追溯 |
| fn 引擎 | MCP tools via iota-fun | 标准化接口，可扩展 |

## 9. 存储设计详解：6 桶记忆的存储结构

### 9.1 各桶特征

| 桶 | 写入频率 | 读取频率 | 数据量 | TTL | 查询模式 |
|----|---------|---------|--------|-----|---------|
| identity | 极低 | 每次 prompt | <50 条/user | 365d | scope=user, facet=identity, 全量读 |
| preference | 低 | 每次 prompt | <100 条/user | 365d | scope=user, facet=preference, 全量读 |
| strategic | 低 | 每次 prompt | <100 条/project | 90d | scope=project, facet=strategic, 全量读 |
| domain | 中 | 每次 prompt | 可达 1000+ 条/project | 90d | scope=project, facet=domain, **FTS/向量** |
| procedural | 中 | 按需 | <200 条/project | 30d | scope=project, type=procedural, FTS |
| episodic | **高** | 当前 session | 高频增长 | 7d | scope=session, type=episodic, 时间序 |

### 9.2 完整 Schema（含多用户预留）

```sql
CREATE TABLE memory (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL CHECK(type IN ('semantic','episodic','procedural')),
  facet TEXT CHECK(facet IN ('identity','preference','strategic','domain')),
  scope TEXT NOT NULL CHECK(scope IN ('session','project','user','global')),
  scope_id TEXT NOT NULL,
  content TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  confidence REAL NOT NULL DEFAULT 1.0,
  source_backend TEXT,
  source_session_id TEXT,
  metadata_json TEXT,
  ttl_days INTEGER NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  expires_at INTEGER NOT NULL,
  supersedes TEXT,
  access_count INTEGER DEFAULT 0,
  last_accessed_at INTEGER,
  -- 多用户预留（当前默认值，代码路径跳过）
  owner TEXT NOT NULL DEFAULT 'local',
  visibility TEXT NOT NULL DEFAULT 'private'
    CHECK(visibility IN ('private','team','public'))
);

-- 核心索引：覆盖 6 桶 recall 查询路径
CREATE INDEX idx_recall_semantic
  ON memory(scope, scope_id, facet, confidence DESC)
  WHERE type = 'semantic';

CREATE INDEX idx_recall_procedural
  ON memory(scope, scope_id, confidence DESC)
  WHERE type = 'procedural';

CREATE INDEX idx_recall_episodic
  ON memory(scope, scope_id, created_at DESC)
  WHERE type = 'episodic';

-- 去重
CREATE UNIQUE INDEX idx_dedup
  ON memory(scope, scope_id, type, facet, content_hash);

-- 过期清理
CREATE INDEX idx_expires ON memory(expires_at) WHERE expires_at IS NOT NULL;

-- FTS5 全文搜索（domain + procedural 桶主要召回方式）
CREATE VIRTUAL TABLE memory_fts USING fts5(
  content,
  content='memory',
  content_rowid='rowid',
  tokenize='unicode61'
);

-- FTS 同步触发器
CREATE TRIGGER memory_ai AFTER INSERT ON memory BEGIN
  INSERT INTO memory_fts(rowid, content) VALUES (new.rowid, new.content);
END;
CREATE TRIGGER memory_ad AFTER DELETE ON memory BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, content) VALUES('delete', old.rowid, old.content);
END;
CREATE TRIGGER memory_au AFTER UPDATE ON memory BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, content) VALUES('delete', old.rowid, old.content);
  INSERT INTO memory_fts(rowid, content) VALUES (new.rowid, new.content);
END;
```

### 9.3 6 桶 Recall 查询模式

```rust
// identity + preference：小量全量读
fn recall_user_semantic(scope_id: &str, facet: &str, limit: u32, min_conf: f64) {
    "SELECT * FROM memory
     WHERE scope='user' AND scope_id=? AND type='semantic' AND facet=?
       AND confidence >= ? AND expires_at > ?now
     ORDER BY confidence DESC, updated_at DESC
     LIMIT ?"
}

// domain：量大，FTS 语义匹配优先
fn recall_domain(scope_id: &str, query: &str, limit: u32) {
    "SELECT m.* FROM memory m
     JOIN memory_fts f ON m.rowid = f.rowid
     WHERE m.scope='project' AND m.scope_id=? AND m.facet='domain'
       AND memory_fts MATCH ?
       AND m.expires_at > ?now
     ORDER BY rank, m.confidence DESC
     LIMIT ?"
}

// episodic：按时间倒序
fn recall_episodic(session_id: &str, limit: u32) {
    "SELECT * FROM memory
     WHERE scope='session' AND scope_id=? AND type='episodic'
       AND expires_at > ?now
     ORDER BY created_at DESC
     LIMIT ?"
}
```

### 9.4 为什么单表而非 6 张表

| 方案 | 优点 | 缺点 |
|------|------|------|
| **单表 + 条件索引** ✅ | 统一 CRUD、migration 简单、FTS 共享、跨桶查询 | 无（单用户数据量 ~900 行） |
| 6 张表 | 各桶独立优化 | schema 重复、migration 复杂、FTS 需 6 个虚表 |

单用户预估总量：identity 20 + preference 50 + strategic 30 + domain 500 + procedural 100 + episodic 200 ≈ **900 行**。SQLite 单表百万行无压力。

### 9.5 向量检索扩展（Phase 3b）

```sql
CREATE TABLE memory_embeddings (
  memory_id TEXT PRIMARY KEY REFERENCES memory(id),
  embedding BLOB NOT NULL,
  model TEXT NOT NULL
);
```

数据量小时 brute-force cosine similarity 即可；domain 桶是向量检索的主要受益者。

## 10. 多用户场景：记忆隔离与共享

### 10.1 可见性矩阵

```
scope=user    + owner=alice             → 只有 alice 可见（identity/preference）
scope=project + owner=alice + private   → alice 的个人项目笔记
scope=project + owner=alice + team      → 团队所有人可见（domain facts）
scope=global  + visibility=public       → 所有人共享（组织级规范）
scope=session + owner=alice             → 永远只有 alice 的会话记录
```

### 10.2 各桶的隔离/共享策略

| 桶 | 隔离需求 | 共享需求 | 策略 |
|----|---------|---------|------|
| identity | 严格隔离 | 永不共享 | owner 过滤 |
| preference | 严格隔离 | 永不共享 | owner 过滤 |
| strategic | 可共享 | 团队共享项目目标 | visibility=team 时全员可见 |
| domain | 可共享 | 团队共享领域知识 | visibility=team 时全员可见 |
| procedural | 可共享 | 团队共享操作规程 | visibility=team 时全员可见 |
| episodic | 严格隔离 | 永不共享 | session 天然单用户 |

### 10.3 存储拓扑：分阶段演进

**阶段 A — 单用户本地（当前目标）：**

```
~/.i6/context/memory.sqlite    ← 一个人的所有记忆
owner = "local" (固定)
visibility 列不参与过滤，代码路径跳过
```

**阶段 B — 团队共享（中期）：**

```
~/.i6/context/memory.sqlite              ← 个人记忆（identity/preference/episodic）
<project>/.iota/shared-memory.sqlite     ← 项目级共享（strategic/domain/procedural）
```

项目级 memory 放在 workspace 内，跟随 git 版本控制。团队成员 pull 即可获取。

**阶段 C — 中心化服务（远期）：**

```
个人记忆 → 本地 SQLite（不变）
团队记忆 → daemon HTTP API → PostgreSQL/Redis
```

### 10.4 多用户 Recall 查询调整

```rust
// 单用户模式（当前）：直接按 scope 查
fn recall_single_user(user_id: &str, project_dir: &str, session_id: &str) {
    identity:   WHERE scope='user' AND scope_id=?user_id AND facet='identity'
    preference: WHERE scope='user' AND scope_id=?user_id AND facet='preference'
    strategic:  WHERE scope='project' AND scope_id=?project_dir AND facet='strategic'
    domain:     WHERE scope='project' AND scope_id=?project_dir AND facet='domain'
    procedural: WHERE scope='project' AND scope_id=?project_dir AND type='procedural'
    episodic:   WHERE scope='session' AND scope_id=?session_id AND type='episodic'
}

// 多用户模式（未来）：增加 visibility 过滤
fn recall_multi_user(user_id: &str, project_dir: &str, session_id: &str) {
    // identity/preference/episodic: 只看自己
    identity: WHERE ... AND owner=?user_id

    // project memory: 自己的 private + 团队的 team + 全局 public
    domain: WHERE scope='project' AND scope_id=?project_dir AND facet='domain'
            AND (owner=?user_id OR visibility IN ('team','public'))

    // episodic: 永远只看自己的 session
    episodic: WHERE ... AND owner=?user_id
}
```

### 10.5 实现策略

| 阶段 | 做什么 | 不做什么 |
|------|--------|---------|
| **现在** | schema 预留 owner+visibility 列，默认 local/private | 不实现多用户查询 |
| **中期** | 项目级 memory 写入 .iota/shared-memory.sqlite，git 共享 | 不搭中心化服务 |
| **远期** | daemon 暴露 memory API，team namespace | — |

### 10.6 SQLite vs Redis 选择理由

| 维度 | SQLite | Redis |
|------|--------|-------|
| Windows 支持 | 原生完美 | 无官方 Windows 版 |
| 安装依赖 | 零（rusqlite 内嵌） | 需额外安装+运行 |
| 全文搜索 | FTS5 内置 | 需 RediSearch 模块 |
| 多用户物理隔离 | 文件级隔离（个人 vs 项目目录） | 需 key namespace |
| 备份/迁移 | 单文件 cp | RDB/AOF |
| 并发写 | daemon 集中持有连接，无冲突 | 天然支持但过度 |
| 适用场景 | 单用户 CLI 工具 ✅ | 分布式多用户服务 |

关键判断：daemon（TCP 47661）已集中所有 engine 访问，不存在多进程并发写 DB 的场景。SQLite 的单写者限制完全不是问题。
